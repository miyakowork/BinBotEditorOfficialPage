# BinBotEditor 官网浅色视觉、动效与产品文案升级设计规格

## 1. 背景与目标

当前 BinBotEditor 官网已经具备中文单页结构、macOS 即将发布状态和 GitHub Pages 部署能力。本次升级不改变产品范围或部署方式，重点解决三个问题：页面视觉表现不够鲜明、现有动效缺少完整叙事节奏、产品文案没有充分反映 BinBotEditor 已有能力。

本规格覆盖官网的视觉、页面结构、动效、中文文案、响应式降级和验证方式。它覆盖原设计规格中的深色“夜幕工作台”方向；其他仍适用的产品边界继续保留。

成功标准：

- 首屏明确传达 BinBotEditor 是一款本地优先的 macOS 编辑器。
- 页面形成可识别的浅色品牌视觉，而不是通用的渐变卡片模板。
- 用户能理解语言支持、离线格式化、搜索导航、Markdown、JSON、会话恢复和可配置体验等真实能力。
- 强动效不牺牲滚动性能、移动端可用性、键盘操作或减少动态效果设置。
- 产品尚未发布时不出现虚假下载地址、猜测日期或无法履行的承诺。

## 2. 已确认的设计方向

采用浅色 A 方案“发光纸面工作台”。页面以暖白纸面为基础，用近黑色文字和深色编辑器窗口建立对比，品牌主色为清晰的紫色，青蓝只用于代码状态与少量交互反馈。

“强视觉表现”不依赖大面积深色背景或堆叠玻璃卡片，而由以下元素共同构成：

- 超大中文标题和紧凑字距。
- 深色编辑器窗口作为贯穿页面的产品视觉锚点。
- 低对比网格、轻微纸张纹理和局部紫蓝光场。
- 巨幅能力数字与产品界面特写交替出现的章节节奏。
- 指针、滚动与产品演示相互配合的分层动效。

推荐色彩语义：

- 页面底色：暖白或浅米白，不使用纯白铺满所有区域。
- 主要文字：近黑紫色，保证长文本可读性。
- 品牌主色：中高饱和紫色，用于重点词、按钮和状态线。
- 辅助色：低饱和青蓝，用于代码高亮、格式化结果和交互反馈。
- 产品窗口：近黑背景，保留真实代码编辑器的沉浸感。

## 3. 信息架构与核心文案

页面继续采用中文单页结构，导航精简为“产品能力”“工作流”“为 Mac 而生”“发布计划”。

### 3.1 首屏

状态标识：

> BinBotEditor for macOS · 首版试用即将发布

主标题：

> 一台 Mac，<br>
> 就是完整工作台。

产品说明：

> 从打开本地文件，到跨项目搜索、离线格式化与 Markdown 实时预览。BinBotEditor 把日常编辑需要的能力，收进一个快速、专注、可自由配置的 macOS 编辑器。

首屏能力标签：

> 30+ 语言支持 · 14 种离线格式化 · 核心编辑无需云端

主操作显示“首版试用即将发布”，并明确为发布状态而非可用下载链接。次操作为“探索产品能力”，滚动到第一个能力章节。若保留 GitHub 入口，文案使用“关注发布进度”，链接指向当前公开仓库，不暗示仓库中已有应用安装包。

### 3.2 能力叙事

功能不再以同权重卡片堆叠，而按编辑流程组织为七个章节：

1. **打开即懂**：支持 30+ 种语言的语法识别、高亮和文件类型匹配。
2. **格式化不出本机**：内置 Prettier、Ruff、ktfmt 等引擎，覆盖 14 种常用语言。
3. **大项目也能快速定位**：全局搜索、区分大小写、分组结果、文件树和多标签协同工作。
4. **Markdown 不必来回切换**：实时预览、文档大纲、GFM 和代码块高亮。
5. **JSON 不只是纯文本**：格式化、深度解析、折叠与展开、转义与反转义、压缩复制。
6. **离开后继续现场**：自动保存、会话恢复与延迟加载标签页。
7. **按你的方式编辑**：快捷键、主题、字号、换行和编辑习惯可配置。

能力数字必须来自实际产品实现，不添加未经验证的速度、体积、用户数或性能倍数。`30+` 指实际语言识别范围，`14` 指已确认的离线格式化语言范围。“核心编辑无需云端”不覆盖可选 AI 能力，避免与可选的 AI 自动命名产生冲突。

### 3.3 macOS 与智能能力

单独设置 macOS 章节，强调本地文件工作流、桌面快捷键和面向 Mac 的交互体验。可选 AI 自动命名作为次要辅助能力出现，使用文案：

> 智能功能适时出现，但不会接管你的工作流。

页面不得把 AI 描述为编辑器运行的必要条件，也不得暗示所有智能能力都离线完成。

### 3.4 发布区

结束标题：

> 把复杂留在编辑器里，<br>
> 把专注留给你。

发布区明确首版仅支持 macOS、试用版本即将发布。未提供真实安装包前，所有发布操作保持一致，不跳转到空地址。FAQ 可继续说明平台、数据位置、格式化方式和发布时间；未知日期统一回答“尚未公布”。

## 4. 页面构成与组件边界

页面按以下职责拆分，命名可根据现有工程习惯调整，但边界保持一致：

- `HeroStage`：首屏标题、发布状态、能力标签和主产品舞台。
- `ProductDemo`：以真实能力为基础，演示打开文件、编辑、格式化和预览流程。
- `CapabilityChapter`：复用章节布局，承载能力数字、说明和产品界面特写。
- `WorkflowStrip`：表现搜索、文件树、多标签和会话恢复之间的连续工作流。
- `MacSection`：集中说明 macOS、本地文件和可选智能能力。
- `ReleaseSection`：提供一致的发布状态与关注入口。
- `MotionSurface`：只负责指针光场和轻微空间倾斜。
- `Reveal`：统一处理章节进入、可视区状态和无观察器降级。
- `MagneticButton`：统一按钮的桌面端磁吸反馈和触屏降级。

产品名称、能力数字、语言与格式化清单、章节文案和发布状态集中在静态内容配置中。组件只负责布局与交互，避免在多个位置复制关键声明。

`ProductDemo` 使用轻量的 HTML/CSS 编辑器示意界面，不依赖远程图片。演示必须对应真实功能，不虚构终端、协作或云同步能力。自动步骤只改变演示界面的选中状态、代码高亮和面板可见性，不修改页面正文。

## 5. 动效系统

### 5.1 首屏入场

导航、状态标识、标题、说明、操作区、能力标签和编辑器窗口按层级依次出现。标题按行进入，编辑器窗口从轻微透视和缩放状态落定。入场完成后不持续移动正文，保证阅读稳定。

### 5.2 指针与悬停

桌面端使用一个全局 `requestAnimationFrame` 控制器，把指针位置写入 CSS 自定义属性。局部紫蓝光场跟随指针，编辑器窗口只产生约 2 至 3 度的空间倾斜。按钮和能力卡片提供轻微磁吸、光边和位移反馈。

指针事件不得直接触发 React 每帧重新渲染。触屏、粗指针和不支持悬停的设备不启用指针光场、磁吸或三维倾斜。

### 5.3 滚动叙事

章节进入视口时依次显现能力数字、正文和界面特写。产品演示在进入可视区后运行，离开可视区后暂停，重新进入时可从当前章节的稳定状态继续。避免多个循环动画同时运行。

巨幅能力数字可做短距离进入或遮罩显现，但不使用无法验证的动态计数。语言标签、代码行和状态提示允许有限的错峰出现，不做无限高速滚动。

### 5.4 性能与减少动态效果

主要动画限定为 `transform` 和 `opacity`。光场使用尺寸受控的背景或伪元素，避免大面积高频模糊。滚动监听使用 `IntersectionObserver`，不在滚动事件中执行布局读取。

系统启用 `prefers-reduced-motion: reduce` 后：

- 内容直接显示最终状态。
- 关闭视差、磁吸、循环演示和空间倾斜。
- 保留不依赖移动的颜色与焦点反馈。

若 `IntersectionObserver`、指针 API 或脚本不可用，正文、按钮和链接仍处于可见且可操作状态。

## 6. 响应式与可访问性

- 桌面端保留完整的光场、透视编辑器和横向章节变化。
- 平板端降低窗口景深、光场范围与位移幅度。
- 手机端改为单列结构，编辑器窗口保持在视口宽度内，只保留淡入和步骤状态切换。
- 中文标题使用流体字号与受控换行，避免孤字、遮挡和横向滚动。
- 导航、按钮、链接和 FAQ 保持语义化结构、可见焦点和合理触控目标。
- 装饰网格、光场和代码背景对辅助技术隐藏；产品能力文字不可只存在于动画或装饰图形中。
- 文本对比度、按钮状态和浅色背景上的紫色用法需要在实际页面中检查，不以视觉稿替代验证。

## 7. 数据流、状态与错误降级

页面没有后端数据流。静态内容配置向各章节提供文案、数字和发布状态；发布状态在一个位置定义，所有入口共用该值。局部状态只包括移动导航、FAQ、演示步骤和当前可视章节，不持久化、不上传。

页面不添加分析脚本、订阅表单、账号、遥测或外部运行时依赖。GitHub Pages 资源继续使用 Vite 的仓库子路径配置。

降级规则：

- JavaScript 未执行时，核心文案、产品能力和发布信息仍可阅读。
- 装饰资源或动画初始化失败时，不隐藏正文。
- 未提供安装包时，发布按钮始终显示即将发布状态。
- 外部 GitHub 链接不可用时，不影响其余页面浏览。
- 未知产品信息不猜测，用明确的未发布状态代替。

## 8. 验证策略

自动验证包括：

- 关键中文文案、`30+`、`14`、macOS 和首版试用状态存在且一致。
- 导航锚点、站内滚动入口与 GitHub 外部链接有效。
- 发布按钮不包含虚假下载地址。
- 减少动态效果、无 `IntersectionObserver` 和触屏降级逻辑可验证。
- GitHub Pages 子路径下的构建资源地址正确。
- 前端测试、TypeScript 检查和 Vite 生产构建通过。

人工浏览器验证覆盖约 1440px 桌面、768px 平板和 390px 手机视口，检查首屏层级、章节节奏、横向溢出、键盘焦点、演示暂停、减少动态效果和控制台错误。部署后再次打开 GitHub Pages 线上地址，确认 HTML、脚本、样式和品牌资源均成功加载。

## 9. 非目标

本次不添加真实安装包、Windows/Linux 版本、多语言切换、在线编辑器、账号系统、邮件订阅、支付、博客、云同步、协作功能或新产品能力。不为追求视觉效果引入 3D 引擎、视频背景或大型动画框架，也不修改 GitHub Pages 的现有部署架构。

## 10. 交付边界

交付物为当前 React/Vite 官网的视觉、动效和文案升级，以及相应测试。实现完成并通过本地验证后推送至公开仓库 `miyakowork/BinBotEditorOfficialPage` 的 `main` 分支，由现有 GitHub Pages 工作流发布到线上网站。
