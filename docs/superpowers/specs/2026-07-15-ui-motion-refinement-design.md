# BinBotEditor 官网 UI 与动效精修设计规格

## 1. 背景与目标

当前官网已经形成暖白纸面、规则网格、超大中文标题、紫色品牌强调和深色编辑器舞台。本轮不重做信息架构、不扩大产品能力范围，也不改变下载与部署逻辑；目标是在现有品牌系统内同时强化三点：首屏冲击力、滚动叙事节奏、产品演示真实感。

用户已在三套视觉探索中选择方案 1，参考图见 [2026-07-15-ui-motion-refinement-reference.png](assets/2026-07-15-ui-motion-refinement-reference.png)。参考图用于约束构图、层级和氛围，实际实现继续以现有内容配置、真实功能与可访问性要求为准。

成功标准：

- 首屏在桌面端形成“超大标题 + 右下抬升编辑器”的鲜明品牌构图。
- 下载操作、能力摘要和产品舞台在一个视口内具有明确主次，不互相争夺注意力。
- 产品演示准确呈现打开文件、专注编辑、离线格式化和实时预览。
- 后续章节不再像同权重卡片列表，而形成连续、可感知的滚动节奏。
- 手机、触控设备、混合输入设备和减少动态效果模式保持稳定可用。

## 2. 设计原则

### 2.1 延续现有品牌

- 保留暖白或浅米白底色、低对比网格、近黑文字、紫色主色和少量青色状态反馈。
- 保留浮动圆角导航与深色编辑器窗口，不引入全页深色、玻璃拟态堆叠、照片或装饰性 3D 物体。
- 标题继续使用具有编辑感的中文展示字体，正文和界面文字使用清晰的无衬线字体；全页最多两套字体。

### 2.2 产品优先

视觉效果必须服务于产品理解。编辑器舞台中的文件树、标签页、代码、Markdown 预览、格式化命令和本地状态均来自现有能力，不展示协作、云同步、终端或其他不存在的功能。

### 2.3 动效有起止

动效负责建立层级和解释状态，不让正文持续漂移。首屏入场完成后保持稳定；循环只存在于处于视口内的产品演示，离开视口、触控设备或减少动态效果模式立即停止。

## 3. 首屏构图

### 3.1 桌面布局

首屏采用非对称编辑式构图：

- 导航保持单条浮动容器，品牌位于左侧，站内锚点和下载操作位于右侧。
- 状态眉题位于标题上方，并与一条短紫色引导线建立起点。
- 主标题保持两行超大中文排版；第一行近黑，第二行的关键部分使用紫色，收紧字距并控制行间重叠感。
- 产品说明、主操作和次操作集中在标题下方左侧，不再分散到多个视觉焦点。
- 三项能力摘要使用轻量分栏和图形标记，成为标题与产品舞台之间的节奏层。
- 编辑器窗口从右下方抬升进入首屏，与标题产生受控重叠，但不得遮挡核心文字或操作。
- 右上区域保留受控紫色光场和细轨迹线，用来连接标题与产品舞台；装饰不承载信息。

### 3.2 响应式

- 平板端降低标题尺寸和编辑器重叠幅度，保持左右关系但减少空间倾斜。
- 手机端改为标题、说明、操作、能力摘要、编辑器的单列顺序；编辑器不与标题重叠。
- 流体字号必须避免孤字、裁切和横向滚动，所有触控操作目标至少保持现有可用尺寸。

## 4. 产品舞台与演示

`ProductDemo` 继续使用真实 HTML/CSS 界面，不依赖远程截图。舞台需要增强以下细节：

- 更清晰的窗口栏、文件树层级、活动标签、代码选择区、Markdown 预览和状态栏。
- 将“打开文件 → 专注编辑 → 离线格式化 → 实时预览”步骤与舞台内部状态一一对应。
- 步骤切换时，只更新相关局部：活动文件、代码强调、格式化反馈、预览面板和步骤状态。
- 当前步骤采用紫色焦点线或青色状态点，非当前步骤降低对比但仍可读。
- 演示在进入视口后运行，离开视口暂停；重新进入时从稳定步骤继续。

不增加可编辑输入、在线运行代码或虚假的下载进度。演示仍是产品能力说明，而不是嵌入式编辑器。

## 5. 后续章节节奏

### 5.1 能力章节

现有能力数据和文案保持集中配置。布局改为“大数字/关键词 + 产品细节 + 说明”的交替结构：

- 语言与格式化章节突出 `30+` 和 `14`，标签作为次级信息环绕或沿基线展开。
- Markdown 与 JSON 章节使用更明确的界面细节，不做同形卡片复制。
- 奇偶章节通过左右布局、留白比例和背景光场轻微变化形成节奏，但不改变阅读顺序。

### 5.2 工作流与 Mac 章节

- 工作流章节强化文件树、搜索、多标签和会话恢复之间的连续关系，使用一条贯穿路径而不是独立卡片。
- Mac 章节继续强调本地文件体验和可选智能能力，视觉上使用简洁的桌面状态与本地标识，不模拟硬件照片。
- FAQ 与页尾保持克制，为前面高密度视觉提供收束。

## 6. 动效系统

### 6.1 首屏入场

导航、眉题、标题两行、说明、操作、能力摘要和编辑器窗口按层级依次进入。标题使用短距离遮罩或上移；编辑器从轻微缩放、透视与下移状态落定。总入场不应形成漫长等待，完成后正文和操作保持静止。

### 6.2 指针反馈

- 延续单一 `requestAnimationFrame` 控制器，通过 CSS 自定义属性驱动光场和约 2 至 3 度的编辑器倾斜。
- CTA 和少量关键入口允许轻微磁吸，不给普通文本或每个标签添加磁吸。
- 指针轨迹线只做缓慢偏移或光点跟随，不绘制高频粒子系统。

### 6.3 滚动显现

- 章节标题、数据、正文和产品细节分为 2 至 3 个层级错峰进入。
- 使用 `IntersectionObserver` 控制显现和演示激活，不在滚动事件中读取布局。
- 主要动画属性限定为 `transform`、`opacity`、遮罩进度和 CSS 变量。

### 6.4 降级

- `prefers-reduced-motion: reduce`：内容直接显示最终状态，关闭循环、视差、磁吸和空间倾斜。
- `hover: none`、`pointer: coarse` 或 `any-pointer: coarse`：关闭指针相关动效和循环演示。
- `IntersectionObserver` 不可用：内容默认可见，产品演示保持稳定状态。
- 动画或脚本失败不得隐藏正文、按钮或下载入口。

## 7. 组件边界与状态

- `Hero`：只负责首屏内容、构图和能力摘要。
- `ProductDemo`：负责舞台标记和演示步骤的视觉映射。
- `useDemoSequence`：负责步骤计时与暂停，不感知具体 DOM。
- `useMotionPreferences`：统一减少动态效果、细指针和触控能力信号。
- `PointerMotion`：只写入指针 CSS 变量，不触发 React 每帧渲染。
- `Reveal`：统一章节进入与无观察器降级。
- `CapabilitySections`、`WorkflowStrip`、`MacSection`：保留现有内容边界，调整布局和动效钩子。
- `styles.css`：扩展现有设计令牌和响应式规则，不引入大型动画依赖。

页面没有新增后端数据流。下载状态、产品声明和能力数字继续由现有配置与组件提供，本轮不修改其业务含义。

## 8. 可访问性与性能

- 确保暖白背景上的紫色正文、青色状态文字和细线达到现有对比度标准；亮色仅用于装饰时不承载语义。
- 保留语义化标题、导航、链接、按钮、FAQ 和可见焦点。
- 装饰网格、光场与轨迹线对辅助技术隐藏。
- 不因动画改变 DOM 阅读顺序，不让能力信息只存在于演示状态。
- 不引入视频、Canvas 粒子、WebGL 或大型动画框架。

## 9. 验证策略

自动验证：

- 现有产品声明、导航、下载入口和演示步骤保持一致。
- 产品演示在可视区外、减少动态效果和触控能力场景下停止循环。
- Reveal 在触控与减少动态效果场景中保持最终可见状态。
- 指针和磁吸逻辑不会在触控事件下残留变换。
- 样式中关键设计令牌、降级查询和对比度契约存在。
- 全量前端测试、TypeScript 编译和 Vite 生产构建通过。

浏览器验证：

- 以 1440 × 1024、768 × 1024 和 390 × 844 检查首屏层级、横向溢出、导航和触控目标。
- 对比选定参考图与实现截图，重点检查标题比例、编辑器位置、留白、按钮层级、边框和圆角。
- 检查产品演示步骤切换、离开视口暂停、滚动显现、键盘焦点和控制台。
- 部署后验证 GitHub Pages 的新资源指纹、桌面与手机加载状态。

## 10. 非目标与交付边界

本轮不添加新功能、账号、订阅、分析、博客、多语言、Windows/Linux、在线编辑、协作或云同步；不改变下载地址、应用签名说明、发布状态或 GitHub Pages 架构。

交付物为当前 React/Vite 官网的 UI 与动效精修、相应测试和线上部署。实现应以选定参考图为视觉目标，同时优先遵守真实产品内容、响应式和可访问性约束。
